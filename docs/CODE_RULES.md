# 项目规则

## 目标

本项目优先保证可读性、可维护性与玩法参数的一致性。规则的核心不是为了“形式统一”，而是为了避免散落逻辑、重复状态和难以追踪的数值来源。

## 代码规范

- 优先复用现有实现与现有常量，不重复造轮子。
- 一个值只保留一个可信来源；不要为了显示或临时修复复制状态。
- 逻辑优先写清楚，再考虑抽象；不要为未来假设场景提前设计。
- 公共逻辑拆到合适模块：
  - 游戏参数放 `src/constants/*`
  - 实体行为放 `src/entities/*`
  - 通用函数放 `src/utils.ts`
  - 运行时编排放 `src/runtime.ts`
  - React 展示层避免承载复杂玩法计算

## 注释规范

- 注释只解释“为什么这样做”，不要复述代码本身。
- 以下场景必须补充注释：
  - 复杂判定或边界条件
  - 不直观的运行时约束
  - 临时兼容逻辑或有明确背景的实现
- 禁止无信息注释，例如“设置 x”“更新数据”这类直接复述代码的内容。

## 命名规范

- 变量、函数使用 `camelCase`。
- 类型、接口、组件使用 `PascalCase`。
- 共享常量、配置对象使用 `UPPER_SNAKE_CASE`。
- 布尔值优先使用 `is`、`has`、`can`、`should` 前缀。
- 事件处理函数优先使用 `handleXxx`。
- 工厂函数优先使用 `createXxx`。
- Jotai atom 统一使用 `xxxAtom`。

## TypeScript 规范

- 禁止显式使用 `any`。
- 优先使用精确类型；无法直接确定时，使用 `unknown` 后再做收窄。
- 公共导出函数的参数与返回值应保持清晰，避免“神秘对象”参数。
- 需要表达结构时，优先定义明确的 `type` 或 `interface`。

## Magic Number 规范

- 禁止在玩法逻辑、判定逻辑、运行时阈值中直接写含义不明的数字。
- 游戏数值优先抽到 `src/constants/*`，并沿用现有 `src/constants/combat.ts` 的组织方式。
- 模块内部只使用一次、但有明确语义的数字，也应提取为模块顶部具名常量。
- 允许直接保留少量基础数字：`0`、`1`、`-1`、`2`，以及明显的数组索引/循环步进值。
- Tailwind 或 JSX 中的尺寸值不做机械式全部抽离；只有在“重复出现”或“具备明确语义”时才提取为 UI 常量。

## 提交前自检

提交前至少执行：

```bash
npm run lint
npm run build
```

如果新增了常量、类型或配置，顺手检查是否已存在同类实现，避免重复定义。
